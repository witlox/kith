# ADR-003: Nostr for Signaling, WireGuard for Transport

## Status: Accepted

## Context

Machines in the mesh need to discover each other and establish encrypted connections. They may be behind NAT, on different networks, or on the move (laptop at home vs. office).

Options considered:
- **Headscale/Tailscale**: excellent UX, requires running a coordination server
- **libp2p**: fully decentralized, complex, heavy dependency
- **Matrix**: rich protocol, high latency for signaling
- **Nostr**: simple signed events on public relays, minimal protocol
- **Plain WireGuard**: works but requires manual peer configuration

## Decision

**Nostr for peer discovery and signaling. WireGuard for encrypted transport.**

- Each kith-daemon publishes a signed Nostr event (kind 30078, parameterized replaceable) containing its WireGuard public key and current endpoint
- Events are tagged with a shared mesh identifier (pre-shared secret)
- Other daemons subscribe to the mesh tag and establish WireGuard tunnels
- When a machine's endpoint changes, it publishes an updated event
- Multiple Nostr relays for redundancy (default: 3+)
- Optional DERP relay for NAT-hostile environments

## Consequences

**Positive:**
- No coordination server to run (sovereignty)
- Nostr relays are public, free, redundant — dozens available
- WireGuard is proven, fast, kernel-level encryption
- Signaling overhead is minimal (a few hundred bytes per peer per update)
- Mesh identifier limits discoverability to authorized machines

**Negative:**
- Public Nostr relays can see that your machines exist (but not traffic content — that's WireGuard)
- NAT hole-punching depends on WireGuard; if it fails, need a DERP relay (which requires hosting something)
- Nostr relays can go down — mitigated by using multiple relays

**Why not Headscale:**
- Requires running and maintaining a coordination server
- Adds an external dependency that can become a single point of failure
- Overkill for a 3-10 machine personal mesh

## Threat Model

Nostr provides **discovery**, not **authentication**. The trust boundaries are:

1. **Nostr relays can see peer events.** Events contain WireGuard public keys and IP endpoints. An observer learns which machines are in your mesh and their network locations. Mitigation: this is metadata only — actual traffic is WireGuard-encrypted and invisible to the relay.

2. **An attacker who knows the mesh identifier can publish fake peer events.** This could inject a rogue endpoint into peer discovery. Mitigation: WireGuard handshake requires the correct private key. A fake peer event pointing to an attacker-controlled IP will fail the WireGuard handshake because the attacker doesn't hold the corresponding private key. Result is denial-of-service (wasted handshake attempt), not interception.

3. **Replay of stale peer events.** An attacker or misbehaving relay serves an old event with a stale endpoint. Mitigation: (a) Nostr kind 30078 (parameterized replaceable) means newer events replace older ones on well-behaving relays. (b) kith-mesh should ignore peer events older than 1 hour. (c) WireGuard handshake fails against wrong endpoint — same as #2.

4. **Mesh identifier is a shared secret.** If leaked, an attacker can monitor discovery. Mitigation: the identifier should be a high-entropy random string (generated by `kith init`), not a human-readable name. Consider encrypting Nostr event content with a key derived from the mesh identifier so that relay observers cannot read WireGuard keys without knowing the identifier.

**Summary:** Nostr is the matchmaking service. WireGuard key verification is the security boundary. A compromised Nostr relay can cause failed connection attempts but cannot intercept traffic or forge mesh membership.

## Validation

Test peer discovery via Nostr from 3+ network configurations (same LAN, across NAT, mobile hotspot). Measure signaling latency (target: <5s per ASM-NET-3). Test relay fallback when Nostr relays are unavailable (ASM-NET-1). Test that fake peer events cause WireGuard handshake failure, not connection to attacker.
